# Tamagotchi Pet on My Device page — Hackathon handoff

Branch: `gkarr-hackathon-tamagotchi`

## What this is

A virtual-pet feature for end users on the My Device page (`/device/:token`). Each
enrolled host gets one pet. Users feed / play / clean / medicate it. Meanwhile
the state of the device itself feeds into the pet's health — failing policies,
disabled disk encryption, and unenrolled MDM all make it sick. Good device
hygiene (all policies passing, disk encryption on) nudges it back toward happy.

Goal: make device hygiene feel personal and fun without shipping a separate
gamification product.

## Architecture decisions

Full-stack (not localStorage). One species for now (`cat`), trivial to add more
later. The pet **cannot die** — all stats are floor-clamped at `1`, so medicine
can always revive. Stats decay on read (no cron); writes happen only on explicit
actions.

### Endpoints

All are device-authenticated via the `token` URL param (Fleet Desktop / device
auth cookie / cert serial — same auth as existing `/device/{token}/*` routes).

| Method | Path | Purpose |
|---|---|---|
| GET  | `/api/_version_/fleet/device/{token}/pet` | Return the pet (or `null` if unadopted). Applies decay + signals on read. |
| POST | `/api/_version_/fleet/device/{token}/pet` | Adopt. Body: `{"name": "...", "species": "cat"}`. |
| POST | `/api/_version_/fleet/device/{token}/pet/action` | Apply an action. Body: `{"action": "feed" \| "play" \| "clean" \| "medicine"}`. |

### Game logic (in `server/service/host_pets.go`)

**Passive decay per hour of neglect** (applied on every read, not persisted):
- `hunger  +3/h` (higher = hungrier)
- `cleanliness −2/h`
- `happiness −1/h`
- `health` does not decay directly — it drains from sustained bad stats.

**Action effects** (on POST /pet/action, persisted):
- Feed: `hunger −30`, `happiness +5`. Over-feeding (current hunger < 20) instead
  does `happiness −10`, `health −2` — stuffed pet.
- Play: `happiness +30`, `hunger +10`.
- Clean: `cleanliness +40`, `happiness +5`.
- Medicine: `health +30`.

**Device-hygiene signals** (applied every read, bounded by stat clamps):
- Failing policies → `−5 health` per failing policy.
- All policies passing → `+2 health`.
- Disk encryption on → `+2 happiness`; off → `−10 happiness`.
- MDM enrollment = `"Off"` → `−10 health`.

**Cross-stat feedback:**
- `hunger ≥ 90` → `−5 health` (starving).
- `cleanliness ≤ 10` → `−5 health` (filthy).

**Derived mood** (priority order):
`sick` (health<30) > `hungry` (hunger>75) > `dirty` (cleanliness<25) >
`sad` (happiness<35) > `happy` (all three positive) > `content`.

## Recommended next steps

### Must-do before merge
1. **Unit tests for the service logic.** `server/service/host_pets_test.go`.
   `applyDecayAndSignals`, `computeMood`, `adjustStat`, and the action switch
   are all pure functions over a `*fleet.HostPet` — easy to table-test. Mock
   the datastore via `mock.DataStore{}`. This is the one place I'd feel bad
   merging without tests because the decay math is easy to typo.
2. **Bump migration timestamp** if anything newer has landed on `main` by the
   time you PR. Use `/bump-migration` or `make migration name=...`.
3. **Regenerate the mock with `mockimpl`** if you have it installed, so future
   diffs don't fight my hand-written entries. The mock file layout matches
   what the generator produces; confirm by running `mockimpl -s DataStore
   -i fleet.Datastore -p server/fleet` (or the project's wrapper command —
   check `server/mock/mockimpl/README.md`) and diffing.
4. **Run `make lint-go` (full lint)** — I only ran `go vet`. The `gosec`,
   `revive`, and `forbidigo` linters may flag things I missed.
5. **Frontend TS compile.** `yarn install && yarn tsc --noEmit` (or whatever
   check the repo uses) to catch any type mismatches I couldn't verify here.
6. **Hook up activity logging?** Optional. Fleet has an activity-log system
   (`server/activity/`). Could emit `pet_adopted`, `pet_fed`, etc. for admin
   visibility / auditing. Probably overkill for hackathon.

### Nice-to-have polish
7. **More species.** Structure is ready — just extend `HostPetSpecies` const
   block, `IsValidHostPetSpecies`, and the `MOOD_EMOJI` map with a per-species
   dimension. Offer species selection in the adoption form.
8. **"Last interacted" hint on the UI.** "Whiskers last saw you 3 hours ago."
   Helps communicate the decay mechanic.
9. **Mood badge on the tab itself.** Put a red dot on the Pet tab when the pet
   is `sick` / `hungry` / `dirty`, the same way Policies shows a failing count.
   One line in `DeviceUserPage.tsx`.
10. **Audio / confetti on successful actions.** Trivial in CSS but enormously
    demo-able.
11. **Pet history / graph.** Lightweight time-series of stats. Would need a
    second table (`host_pet_events`) — leave until after demo.
12. **Reset / re-adopt flow.** Currently once adopted, you're stuck with that
    name + species. Add a "retire" button that deletes the row so the user can
    adopt again.
13. **Rate-limit the action endpoint.** Nothing stops a user from spam-feeding
    to cap happiness. The `errorLimiter` middleware is IP-based and pretty
    coarse. For a real ship: add a simple per-host cooldown (e.g. last action
    within 10s → 429). Not urgent — the over-feeding debuff already punishes
    spam-feeding, and other actions don't cause similar harm.
14. **Mobile view.** Currently the mobile/iOS path of `DeviceUserPage` only
    renders `<SelfService>`. The pet tab is desktop-only today. If this goes
    past hackathon, either add the pet to the mobile path or gate it cleanly.
15. **Tab ordering pre-existing issue.** `DeviceUserPage` has a latent bug
    where `tabPaths` doesn't always line up with the rendered `<Tab>` count
    when `isSoftwareEnabled` is false on free tier. Unrelated to pets, but
    worth noting if you hit weird tab-index routing issues on non-premium,
    non-software-enabled deployments.

## Open questions to pick up on
1. **Species selection UX.** When you add a second species, do you want a
   dropdown on adoption, or a "mystery egg" that hatches into a random species?
2. **Compliance gamification — how far?** The current signals (policies, disk
   encryption, MDM) are the low-hanging fruit. Do we want to include software
   updates (`host.os_version` vs. some "latest" reference)? Password manager
   installed? This is where the feature can either stay cute or become an
   actual compliance nudge tool.
3. **Free tier visibility.** Right now the Pet tab shows on both premium and
   free tiers. Is that what you want, or should this be a premium differentiator?
4. **What happens to the pet when the host is transferred between fleets?**
   Currently: nothing. The pet travels with the host (same `host_id`). Seems
   fine — confirm.
