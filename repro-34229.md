# Reproduction: #34229 — Missing label validation errors in GitOps

## Bug

`fleetctl gitops` silently accepts labels with invalid combinations of parameters instead of showing errors.

## How to reproduce

Create a `default.yml` GitOps config with the following labels section and run `fleetctl gitops -f default.yml`:

### Case 1: Manual label with criteria, query, and platform

Manual labels should only accept `name`, `description`, and `hosts`. Specifying `criteria`, `query`, or `platform` should be an error.

```yaml
labels:
  - name: Bad manual label
    label_membership_type: manual
    query: SELECT 1
    platform: darwin
    criteria:
      vital: end_user_idp_group
      operator: "="
      value: Engineering
    hosts:
      - host1
```

**Expected:** Error — manual labels cannot have `query`, `platform`, or `criteria`.
**Actual:** Silently accepted.

### Case 2: Dynamic label with criteria

Dynamic labels use `query`, not `criteria`.

```yaml
labels:
  - name: Bad dynamic label
    label_membership_type: dynamic
    query: SELECT 1
    criteria:
      vital: end_user_idp_group
      operator: "="
      value: Engineering
```

**Expected:** Error — dynamic labels cannot have `criteria`.
**Actual:** Silently accepted.

### Case 3: Host_vitals label with query and platform

Host_vitals labels use `criteria`, not `query` or `platform`.

```yaml
labels:
  - name: Bad host_vitals label
    label_membership_type: host_vitals
    query: SELECT 1
    platform: darwin
    criteria:
      vital: end_user_idp_group
      operator: "="
      value: Engineering
```

**Expected:** Error — host_vitals labels cannot have `query` or `platform`.
**Actual:** Silently accepted.

### Bonus: Invalid platform value

```yaml
labels:
  - name: Bad platform label
    label_membership_type: dynamic
    query: SELECT 1
    platform: invalidplatform
```

**Expected:** Error — `invalidplatform` is not a valid platform.
**Actual:** Silently accepted at parse time (only validated server-side in `ApplyLabelSpecs`).

## Root cause

The GitOps YAML parser at `pkg/spec/gitops.go` (in the `parseLabels` function) only validates:
- Name is present
- Non-manual labels have query or criteria
- Name is ASCII
- Host vitals criteria is structurally valid

It does **not** check for mutually exclusive field combinations (query vs criteria vs hosts) or validate platform values. The server-side `ApplyLabelSpecs` in `server/service/labels.go` has some of these checks but not all cross-field exclusivity checks.

## Fix

Added validation in `pkg/spec/gitops.go` `parseLabels()` to reject:
1. Manual labels with `query`, `criteria`, or `platform`
2. Dynamic labels with `criteria` or `hosts`
3. Host_vitals labels with `query`, `platform`, or `hosts`
4. Invalid platform values (checked against `fleet.ValidLabelPlatformVariants`)

Also added matching validation in `server/service/labels.go` `ApplyLabelSpecs()` for defense-in-depth.
